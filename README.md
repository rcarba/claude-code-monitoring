# Claude Code — Token & Cost Monitoring

Self-contained stack to analyse where your Claude Code tokens go: parses the
JSONL transcripts at `~/.claude/projects/`, ingests them into SQLite and
serves a Grafana dashboard.

## Quick start

```bash
git clone https://github.com/rcarba/claude-code-monitoring.git
cd claude-code-monitoring
cp .env.example .env          # optional — adjust TZ, port, etc.
docker compose up -d --build
```

Open <http://localhost:3000>. Anonymous Admin is enabled, no login. The
dashboard "Claude Code — Token & Cost" is provisioned automatically.

> **First run on a new machine:** if you have never used Claude Code, the
> directory `~/.claude/projects/` does not exist yet. The ingester will keep
> retrying every 5 minutes; once you actually run Claude Code, data will start
> showing up on the next tick. No restart needed.

### Platforms

- **Linux & macOS**: works out of the box.
- **Windows**: use [WSL2](https://learn.microsoft.com/en-us/windows/wsl/install).
  Run Docker Desktop with WSL backend, clone this repo *inside* WSL, and use
  the WSL filesystem (`/home/<user>/.claude/projects`). Trying to mount
  `C:\Users\...\.claude` directly is finicky.

### Configuration

Edit `.env` (copy from `.env.example`):

| Var                    | Default                  | What it does                              |
|------------------------|--------------------------|-------------------------------------------|
| `TZ`                   | `Europe/Madrid`          | Timezone for hour-of-day / day-of-week    |
| `GRAFANA_PORT`         | `3000`                   | Host port for Grafana                     |
| `CLAUDE_PROJECTS_DIR`  | `${HOME}/.claude/projects` | Where your JSONL transcripts live       |
| `INTERVAL_SECONDS`     | `300`                    | Ingest loop period                        |

## What it captures

For every assistant message in every JSONL transcript:

| Column          | Meaning                                                 |
|-----------------|---------------------------------------------------------|
| `input_tokens`  | Non-cached input tokens                                 |
| `output_tokens` | Tokens generated by Claude                              |
| `cache_5m`      | Cache writes with 5-min TTL (1.25× input price)         |
| `cache_1h`      | Cache writes with 1-hour TTL (2× input price)           |
| `cache_read`    | Cache reads (0.1× input price — the cheap one)          |
| `cost_usd`      | Cost computed from LiteLLM prices                       |
| `project`       | Folder name extracted from the transcript path          |
| `model`         | e.g. `claude-opus-4-7`, `claude-sonnet-4-6`             |
| `session_id`    | One Claude Code session                                 |

## Panels

- **KPIs**: total cost, total tokens, sessions, cache hit ratio for the period.
- **Cost over time** stacked by project (1 h buckets).
- **Tokens over time** broken down by type (input/output/cache_create/cache_read).
- **Cost by hour of day** to see when you spend the most.
- **Cost by project / by model** (donut).
- **Cost by day of week**.
- **Top sessions by cost** (table).

## Where to look first when optimising

1. **Cache hit ratio < 70%** → you are losing money on cache writes that don't
   pay off. Consider longer rounds in the same session to amortise the
   `cache_5m` cost.
2. **Top project by cost** → that project either has long sessions or
   tool-heavy prompts. Check oversized contexts (huge `CLAUDE.md`, directories
   where `Read` returns a lot).
3. **High `cache_1h`** → 1-hour cache writes cost 2× input. If you are not
   actually reusing them after >5 min, drop them down to 5 m.

## Architecture

```
~/.claude/projects/   →  ingester (Python, every 5 min)  →  /data/tokens.db  →  Grafana (SQLite plugin)
```

- **ingester** (Python 3.12): re-parses any new JSONL files, dedup by
  `message_id`. Idempotent. Reads env: `INTERVAL_SECONDS`,
  `CLAUDE_PROJECTS_DIR`, `DB_PATH`, `TZ`.
- **Pricing**: `pricing.py` fetches the LiteLLM model price file on first use
  and falls back to a hardcoded table if offline.

## Operations

```bash
docker logs -f token-ingester                       # tail ingester
docker compose run --rm ingester python /app/ingest.py --once   # manual run
docker compose down -v                              # full reset (drops DB)
```
